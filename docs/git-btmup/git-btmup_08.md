# 以任何其他名称的提交……

> 原文：[`jwiegley.github.io/git-from-the-bottom-up/1-Repository/6-a-commit-by-any-other-name.html`](http://jwiegley.github.io/git-from-the-bottom-up/1-Repository/6-a-commit-by-any-other-name.html)

理解提交是掌握 Git 的关键。当你心中只有提交拓扑，而将分支、标签、本地和远程仓库等的混乱抛在脑后时，你就知道你已经达到了分支智慧的禅宗境界。希望这种理解不需要砍掉你的手臂——尽管我现在可以理解你为什么考虑过这一点。

如果提交是关键，那么提交的命名就是掌握的门户。有许多命名提交、提交范围以及提交持有的某些对象的方法，这些方法被 Git 的大多数命令所接受。以下是一些更基本用法的总结：

+   **branchname** — 如前所述，任何分支的名称只是该“分支”上最新提交的别名。这和在检出该分支时使用单词 HEAD 是相同的。

+   **tagname** — 标签名别名在命名提交方面与分支别名相同。两者之间的主要区别是，标签别名永远不会改变，而分支别名每次将新提交检入该分支时都会改变。

+   **HEAD** — 当前检出的提交始终被称为 HEAD。如果你检出特定的提交——而不是分支名称——那么 HEAD 仅指该提交，而不是任何分支。请注意，这种情况有些特殊，被称为“使用分离的 HEAD”（我确信这里可以讲一个笑话……）。

+   **c82a22c39cbc32…** — 提交可以使用其完整的 40 字符 SHA1 哈希 ID 进行引用。通常这发生在复制粘贴时，因为通常有其他更方便的方式来引用相同的提交。

+   **c82a22c** — 你只需要使用足够多的哈希 ID 数字，以便在仓库内进行唯一引用。大多数情况下，六位或七位数字就足够了。

+   **name^** — 任何提交的父提交都使用上箭头符号引用。如果一个提交有多个父提交，则使用第一个。

+   **name^^** — 可以连续应用箭头。此别名指的是给定提交名称的“父父”提交。

+   **name²** — 如果一个提交有多个父提交（例如合并提交），你可以使用`name^n`来引用第*n*个父提交。

+   **name~10** — 可以使用波浪号(~)后跟序数来引用提交的第*n*代祖先。这种用法在`rebase -i`等操作中很常见，例如，表示“显示一些最近的提交”。这等同于 name^^^^^^^^^^。

+   **name:path** — 要引用提交内容树中的某个特定文件，请在冒号后指定该文件的名称。这对于显示或显示提交文件的两个版本之间的差异很有帮助：

```sh
  $ git diff HEAD¹:Makefile HEAD²:Makefile

```

+   **name^{tree}** — 你可以仅引用提交持有的树，而不是提交本身。

+   **name1..name2** — 这些别名表示 *提交范围*，这对于使用如 log 等命令查看特定时间段内发生的事情非常有用。左侧的语法指的是从 **name2** 追溯到，但不包括 **name1** 的所有可到达的提交。如果省略 **name1** 或 **name2**，则使用 HEAD 替代。

+   **name1…name2** — “三个点”范围与上面的两个点版本有很大不同。对于 log 等命令，它指的是由 **name1** 或 **name2** 引用的所有提交，但不由两者同时引用。结果是两个分支中所有唯一提交的列表。对于 `diff` 等命令，所表示的范围是在 **name2** 和 **name1** 和 **name2** 的共同祖先之间。这与 `log` 的情况不同，因为 **name1** 引入的更改不会显示。

+   **master..** — 这种用法等同于“`master..HEAD`”。我将其添加在这里，即使它在上文已经暗示，因为我经常在审查对当前分支所做的更改时使用这种类型的别名。

+   **..master** — 这在您执行了 `fetch` 操作并想查看自上次 `rebase` 或 `merge` 以来发生了什么更改时特别有用。

+   **–since=“2 weeks ago”** — 指的是自某个日期以来的所有提交。

+   **–until=”1 week ago”** — 指的是所有在某个日期之前的提交。

+   **–grep=pattern** — 指的是所有提交信息匹配正则表达式模式的提交。

+   **–committer=pattern** — 指的是所有提交者匹配该模式的提交。

+   **–author=pattern** — 指的是所有作者匹配该模式的提交。提交的作者是指创建它所代表更改的人。对于本地开发，这始终与提交者相同，但当通过电子邮件发送补丁时，作者和提交者通常不同。

+   **–no-merges** — 指的是范围内只有一个父提交的所有提交 — 即它忽略了所有合并提交。

大多数这些选项可以混合使用。以下是一个示例，它显示了以下日志条目：对当前分支（从 master 分支创建）所做的更改，由我自己在过去的一个月内完成，包含文本“foo”：

```sh
$ git log --grep='foo' --author='johnw' --since="1 month ago" master..

```
