# 仓库：目录内容跟踪

> 原文：[`jwiegley.github.io/git-from-the-bottom-up/1-Repository/1-directory-content-tracking.html`](http://jwiegley.github.io/git-from-the-bottom-up/1-Repository/1-directory-content-tracking.html)

如上所述，Git 所做的是相当基础的：它维护着目录内容的快照。其内部设计的大部分都可以通过这个基本任务来理解。

Git 仓库的设计在许多方面反映了 UNIX 文件系统的结构：一个 *文件系统* 从根目录开始，通常包含其他目录，其中大多数有叶节点，或 *文件*，它们包含数据。关于这些文件内容的元数据既存储在目录中（名称），也存储在引用这些文件内容的 i-nodes 中（大小、类型、权限等）。每个 *i-node* 都有一个唯一数字来标识其相关文件的内容。虽然你可能有很多目录条目指向特定的 i-node（即硬链接），但 i-node 才是“拥有”存储在文件系统上的内容的。

在内部，Git 具有非常相似的结构，尽管有一两个关键的区别。首先，它通过 *blob* 来表示你的文件内容，这些 *blob* 也在一个称为 *tree* 的类似目录的结构中作为叶节点。正如 i-node 是通过系统分配的唯一数字来唯一标识一样，blob 是通过计算其大小和内容的 SHA1 哈希 id 来命名的。从所有目的和意图来看，这只是一个任意的数字，就像 i-node 一样，但它有两个额外的属性：首先，它验证 blob 的内容永远不会改变；其次，相同的内 容总是由相同的 blob 来表示，无论它出现在哪里：提交之间、仓库之间——甚至整个互联网上。如果多个 tree 引用了同一个 blob，这就像硬链接一样：只要至少还有一个链接指向它，blob 就不会从你的仓库中消失。

Git blob 与文件系统的文件之间的区别在于，blob 不存储关于其内容的元数据。所有此类信息都保存在包含 blob 的 tree 中。一个 tree 可能知道这些内容是一个名为“foo”的文件，该文件于 2004 年 8 月创建，而另一个 tree 可能知道相同的内容是一个名为“bar”的文件，该文件是在五年后创建的。在一个正常的文件系统中，具有相同内容但具有如此不同元数据的两个文件始终会被表示为两个独立的文件。为什么会有这种区别？主要原因是文件系统是为了支持可更改的文件而设计的，而 Git 则不是。Git 仓库中数据不可变的事实使得所有这一切都成为可能，因此需要不同的设计。结果证明，这种设计允许更紧凑的存储，因为所有具有相同内容的对象都可以共享，无论它们在哪里。
